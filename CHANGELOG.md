# CHANGELOG

## v2.2.0 — userType 의도 복원 + description 간결화

- **`anonymous` 재정의(원의도 복원):** "비실명 **계정**(계정별로 구분되는 사내 id 보유)" 로
  되돌리고 `userId`(문자열) **필수**로 변경. v2.0/v2.1 에서 anonymous 를 "합산 사용량(userId
  null)" 으로 좁혔던 것을 바로잡음.
- **`unclassified` 명확화:** userId 가 필수가 아닌 채로 수집되는 **미매핑 사용량** → `userId`
  null, 모델 단위 합산. (= identified/anonymous 는 userId 필수, unclassified 만 null)
- **if/then 갱신:** anonymous→userId 문자열 필수로 변경(이전엔 null 강제였음).
- **description 전면 간결화:** "수집기 측이 추론 서비스에게 구현을 요청"하는 데 필요한
  정보만 남기고 배경설명·가격 수치·OTel 어댑터 등 곁가지 제거.

## v2.1.0 — Codex 리뷰 반영 (스키마 강제력 보강)

v2.0.0 은 "설명은 강하지만 스키마는 느슨한" 부분이 있었다. v2.1.0 은 그 계약들을
**OpenAPI 스키마로 실제 강제**하도록 조였다. 동작 자체의 큰 변화는 없고, 구현체/수집기
간 해석 차이를 없애는 정리다.

| 항목 | v2.0.0 | v2.1.0 | 왜 |
|---|---|---|---|
| 미확정 표현 | `200 + status:partial` 과 `409` 두 갈래 | **`409` 단일화** (`status`/`DataStatus` 삭제, `200`=확정만) | 미확정 신호가 둘이면 수집기가 적재/재시도 판단에서 갈림 |
| `model` | 선택 필드("생략 가능") + "unknown 권장" 혼재 | **required** (+`minLength:1`), 미상은 `"unknown"` | 일부 행만 model 있으면 이중 집계. 메시지도 한쪽으로 정리 |
| `userId` ↔ `userType` | 설명만 ("identified면 존재") | **`allOf` if/then 으로 강제** (identified→문자열 필수, anonymous/unclassified→null) | 논리 키·distinct 계산이 구현마다 달라지는 것 방지 |
| cache 필드 | record optional / summary required (불일치) | **양쪽 optional+default 0 일치** | 혼합 환경 정책 일관성 (vLLM 등은 0/생략) |
| `generatedAt` | `date-time` (offset 허용) | **UTC `Z` suffix 강제(pattern)** | 시각 해석 혼선 제거 |
| summary `date` | 검증 문구 짧음 | `/v1/usage` 와 동일 문구 | 두 엔드포인트 날짜 정책 일치 |
| (문서) | 인라인 주석에 `401` 오기 | 제거 (인증 미요구 유지) | 실제 응답에 401 없음 |

검증: 예시 + 음성(negative) 케이스를 JSON Schema(Draft7)로 통과 확인
(identified+null 거부, anonymous+string 거부, 빈 model 거부, `Z` 없는 timestamp 거부 등).

## v2.0.0 — v1 대비 변경사항 (쉽게 설명)

v2는 새 기능을 넣었다기보다, **"숫자가 조용히 틀어지거나 사라지는 경우"를 전부 막은 버전**이다.
캐시 토큰(정확성), 미확정 구분(유실 방지), ID 분리(집계 안정) — 이 셋이 핵심이고,
나머지는 그걸 보강하는 안전장치다.

각 항목은 **"v1의 문제 → v2의 변경 → 왜 그렇게 결정했나"** 순서로 설명한다.

---

### 1. 캐시 토큰 필드 추가 (가장 큰 변화)

- **v1:** `inputTokens`, `outputTokens` 둘뿐.
- **문제:** 요즘 LLM은 "캐시"를 쓴다. 같은 내용을 또 보내면 캐시에서 읽어오는데(`cacheRead`),
  이건 일반 input보다 **약 10배 싸다**. 반대로 캐시에 처음 저장할 땐(`cacheCreation`) 더 비싸다.
  v1은 이를 구분 못 해서, A서비스는 캐시 토큰을 input에 합쳐 보내고 B서비스는 빼고 보내면
  **두 서비스의 숫자를 비교조차 할 수 없게** 된다.
- **v2:** `cacheReadTokens` / `cacheCreationTokens` 를 추가하고, "`inputTokens`는 캐시 읽기를
  빼고 센다"고 못박았다. 단, 혼합 환경이므로 **선택 필드(기본 0)** 로 둬서 vLLM처럼 캐시 개념이
  없는 서비스는 0으로 두면 된다. `outputTokens`는 reasoning/thinking 토큰을 포함한다.
- **왜:** "토큰 사용량"이라는 단어의 의미를 모든 서비스가 똑같이 쓰도록 강제하기 위해서.
  안 그러면 대시보드 합계 자체가 거짓말이 된다.

> 참고: cache read/creation 단가가 다른 건 사실상 Anthropic 류의 explicit caching 모델이다.
> OpenAI 경유는 cacheRead만, vLLM 자체 호스팅은 둘 다 0(전부 inputTokens)으로 보고하면 된다.

---

### 2. "데이터 아직 안 나옴" vs "사용량 0" 구분

- **v1:** 데이터가 없으면 `404` 또는 빈 응답 — 둘 중 아무거나 써도 됐다.
- **문제:** 수집기가 매일 새벽 2시에 "어제 데이터 줘"라고 호출하는데, 그 서비스의 집계 배치가
  늦어서 아직 준비가 안 됐다면? v1에서는 빈 응답이 온다. 수집기는 이걸 보고
  **"어제 아무도 안 썼구나(사용량 0)"** 라고 오해하고 영구 기록해버린다.
  실제로는 데이터가 늦은 것뿐인데 → **사용량이 조용히 사라진다.**
- **v2:**
  - 진짜 사용량 0 → `200` + 빈 목록 + `status: final` ("확정됐고, 0이 맞아")
  - 아직 준비 안 됨 → `409 data_not_ready` ("나중에 다시 와") + `Retry-After` 안내
- **왜:** "없음"에는 두 종류가 있고, 수집기가 둘을 구분해야 데이터 유실을 막을 수 있어서.
  0과 미확정을 같은 신호로 두면 반드시 사고가 난다.

---

### 3. 서비스 이름을 "ID"와 "표시명"으로 분리

- **v1:** `service: "Code Mate"` 같은 자유 문자열 하나.
- **문제:** 대시보드가 이 이름으로 묶어서(GROUP BY) 합계를 낸다. 그런데 누가
  `"Code Mate "`(뒤 공백)라고 보내거나 나중에 표시명을 바꾸면 **다른 서비스로 인식돼 합계가
  쪼개진다.** (실제로 v1 예시에 공백이 들어간 버그가 있었다.)
- **v2:** `serviceId: code-mate`(절대 안 바뀌는 키) + `serviceGroup: "Code Mate"`(보여주는 이름)
  로 분리. 묶을 땐 ID로, 화면 표시는 이름으로. (서비스 그룹도 `serviceGroupId`로 동일 적용)
- **왜:** 집계의 기준이 되는 값은 "절대 안 변하고 흔들리지 않아야" 하기 때문.
  사람이 보는 예쁜 이름과, 기계가 묶는 키는 역할이 다르다.

---

### 4. `distinctUsers` → `distinctIdentifiedUsers` + "더하지 마세요" 경고

- **v1:** "고유 사용자 수" 필드.
- **문제:** 같은 사람이 서비스 A도 쓰고 B도 쓰면, A의 사용자 수 + B의 사용자 수로 그룹 합계를
  내면 **그 사람이 2명으로 중복 집계**된다.
- **v2:** 이름을 명확히 하고, "이 값은 서비스끼리 더할 수 없는(non-additive) 값"이라고 스펙에
  박았다. anonymous/unclassified는 정의상 distinct 계산이 불가하므로 제외.
- **왜:** 더하면 안 되는 숫자를 더하는 건 흔한 실수라, 아예 문서에서 막아두는 게 안전해서.

---

### 5. `model` 필드 규칙 ("섞지 마세요")

- **v1:** `model`은 선택 필드, 생략하면 "모델 무관 합산".
- **문제:** 같은 사용자에 대해 어떤 행은 model을 적고 어떤 행은 안 적으면, 수집기가
  **토큰을 두 번 더해버릴** 수 있다.
- **v2:** "한 응답 안에서는 model을 **전부 적든가, 전부 빼든가** 하나로 통일.
  모델을 모르면 `"unknown"`이라고 명시." 로 규칙화.
- **왜:** 이중 집계는 조용히 일어나는 가장 위험한 버그라서, 애매한 여지를 없앴다.

### 6. 기타 다듬기

| 항목 | v1 | v2 | 왜 |
|---|---|---|---|
| 신선도 | 없음 | `generatedAt`(집계 시각) 추가 | "이 숫자가 언제 만든 건지" 알아야 신뢰 판단 가능 |
| 정합성 | 불명확 | summary 합 = 상세 합 일치 명시 | 요약 카드와 상세 화면이 어긋나지 않게 |
| 날짜 검증 | 없음 | 오늘/미래 날짜는 `400`으로 거절 | 아직 안 끝난 날을 묻는 건 오류니까 |
| 에러 응답 | `400`만 | `409`/`429`/`500`/`503` + 재시도 안내 | 수집기가 "재시도할지/포기할지" 판단 근거 |
| 페이지 커서 | 만료 가능성 모호 | 만료 안 되는 keyset 방식 권장 | 수집 도중 커서 끊기면 데이터 누락되니까 |
| 토큰/요청 출처 | 불명확 | provider 보고 usage 기준, `requests`=provider 호출 수 | 서비스 간 숫자 비교 가능하게 |
| 인증 | 없음(유지) | 네트워크 ACL 권장 문구 | 사용자별 사용량은 민감 정보라 최소한의 가드 |
